Proyecto: AvisNetwork / ProyectoIntermodularDAM
Entorno: Java 17 / Lubuntu VM
Framework: Spring Boot v3.4.0 (Arquitectura Reactiva)

1. TECNOLOGÍAS PRINCIPALES Y SU FUNCIÓN
------------------------------------------------------------------------
* Java 17 + Spring Boot: El motor principal del servidor. Proporciona la infraestructura y la inyección de dependencias.
* Spring WebFlux (Netty): Servidor web asíncrono y reactivo (Puerto 8080). A diferencia del Spring tradicional (MVC), WebFlux no bloquea los hilos de ejecución, permitiendo manejar miles de peticiones simultáneas con muy pocos recursos.
* Lombok: Herramienta que genera automáticamente código repetitivo (Getters, Setters, Builders, Constructores) durante la compilación, manteniendo las clases del modelo (ej. Player.java) limpias y fáciles de leer.
* Maven: Gestor de dependencias y empaquetador del proyecto (lee el archivo pom.xml).


2. PERSISTENCIA Y BASE DE DATOS
------------------------------------------------------------------------
* Supabase (PostgreSQL): Base de datos relacional alojada en la nube. Guarda la información persistente (Usuarios, Pájaros, Avistamientos, Bandadas, Posts).
* Supavisor (Connection Pooler): Intermediario configurado en Supabase (Puerto 5432) que permite conexiones a la base de datos a través de redes IPv4, solucionando los bloqueos de enrutamiento IPv6.
* Spring Data R2DBC: Driver de conexión de base de datos REACTIVO. A diferencia de JDBC/JPA, R2DBC no bloquea el servidor mientras espera que la base de datos responda.


3. CACHÉ Y RENDIMIENTO (DATOS EN MEMORIA)
------------------------------------------------------------------------
* Redis: Base de datos en memoria (clave-valor). Se utiliza para operaciones que requieren velocidad extrema (ej. tablas de clasificación, datos de sesión, etc.).
* Redisson + Spring Boot Data Redis: Cliente avanzado de Java para interactuar con Redis de forma reactiva y gestionar la caché de la aplicación.


4. TIEMPO REAL Y EMPAREJAMIENTO (SISTEMA DE JUEGO)
------------------------------------------------------------------------
* RabbitMQ (AMQP): Sistema de intermediación y colas de mensajes. Se utiliza para el "MatchmakingService" (emparejamiento de jugadores). Un jugador se encola, RabbitMQ retiene la petición y la empareja cuando encuentra otro jugador disponible.
* RSocket: Protocolo de comunicación bidireccional (Puerto 7000). Mantiene una conexión constante y fluida entre los jugadores y el servidor para gestionar las "Batallas" en tiempo real sin el retardo (latencia) de las peticiones HTTP normales.


5. SEGURIDAD Y AUTENTICACIÓN
------------------------------------------------------------------------
* Spring Security: Filtra todas las peticiones entrantes para asegurar que solo los usuarios permitidos accedan a ciertos endpoints.
* JWT (JSON Web Tokens - jjwt): Sistema de credenciales. Al loguearse/registrarse, el servidor firma un token secreto. El teléfono guarda este token y lo envía en sus próximas peticiones para demostrar quién es sin tener que mandar la contraseña constantemente.
* BCrypt (BCryptPasswordEncoder): Algoritmo de encriptación unidireccional. Las contraseñas de los usuarios (ej. 'AleAdriRober') no se guardan en texto plano en la base de datos, sino como un "Hash" indescifrable por seguridad.


6. INFRAESTRUCTURA DE RED
------------------------------------------------------------------------
* Tailscale: Red privada virtual (VPN). Permite que el servidor Lubuntu, la base de datos, los servicios como Rabbit/Redis y el teléfono móvil se comuniquen de forma segura como si estuvieran todos conectados al mismo router físico (IPs 100.x.x.x).


========================================================================
              FLUJO DE COMUNICACIÓN (CÓMO HABLAN ENTRE SÍ)
========================================================================

Escenario de Ejemplo: Registro de un Jugador

1. El Teléfono (Cliente): El usuario rellena sus datos en la App y el teléfono envía un JSON por HTTP POST a la ruta '/api/auth/register' a través de la red de Tailscale.
2. WebFlux + Security: El servidor Netty recibe la petición, Spring Security comprueba que esta ruta es pública (no exige token previo) y la deja pasar.
3. AuthController (Controlador): Recibe el JSON, extrae "username" y "password" y se los pasa al cocinero (AuthService).
4. AuthService (Lógica de Negocio): 
   - Comprueba si el usuario ya existe llamando a PlayerRepository.
   - Encripta la contraseña usando BCrypt.
   - Crea un objeto 'Player' en memoria indicando que es un registro nuevo (isNewRecord = true).
5. PlayerRepository (R2DBC): Coge el objeto 'Player', genera una sentencia SQL [INSERT INTO players...] y la manda a Supabase (vía el Session Pooler IPv4).
6. Supabase: Guarda los datos en disco y responde al servidor confirmando la creación.
7. JwtUtil: El servidor genera un token JWT firmado.
8. Respuesta Final: El AuthService empaqueta los datos del jugador + el Token y el AuthController lo devuelve al Teléfono con un código HTTP 201 (Created).

Nota sobre el Juego: 
Para las batallas, en lugar de usar peticiones HTTP cortas, el Teléfono abrirá un "túnel" RSocket (Puerto 7000). El servidor enviará al jugador a la "sala de espera" de RabbitMQ. Cuando haya dos, el servidor les notificará en tiempo real por el túnel RSocket para empezar la partida.
